

# 软件名称：sbssh

**V1.0 (本地增强版)** 和 **V2.0 (云同步版)**。


**安卓 VPS SSH/SFTP 管理工具 ( sbssh) 开发文档**

**总策略：** 分阶段开发。v1.0 专注于本地安全管理；v2.0 在 v1.0 基础上平滑升级，引入端到端加密云同步。

# ---

**第一阶段：VMan v1.0 (本地增强版)**

**目标：** 开发一款单机运行、数据完全本地化的 VPS 管理工具，核心是实现安全的本地数据存储和全功能的 SSH/SFTP 体验。

## **1\. 核心功能規格 (v1.0)**

* **VPS 列表管理：** 本地数据库增、删、改、查服务器信息。  
* **多模式连接：**  
  * **密码认证：** 加密存储密码。  
  * **密钥认证：** 支持导入私钥文件（OpenSSH 格式），支持输入密钥密码（Passphrase）。  
* **功能集成：**  
  * **SSH 终端：** 完整的 Linux 终端体验，支持快捷辅助栏。  
  * **SFTP 文件管理器：** 远程文件浏览、上传、下载、删除、重命名、权限修改。  
* **本地安全性 (核心)：**  
  * **应用锁：** 可选生物识别（指纹/面部）解锁应用。  
  * **本地数据加密：** 采用字段级 AES-GCM 加密（密码/私钥等敏感字段加密存储），数据库使用 Room（明文字段保持最小化）。

## **2\. 安全架构设计 (v1.0 \- 为 v2.0 铺路)**

虽然是本地使用，但为了确保 v2.0 升级时用户不需要重新输入所有服务器密码，我们**不使用** Android Keystore 存储密钥，而是采用类似 Bitwarden 的**主密码 (Master Password)** 模型。

### **2.1 密钥衍生模型**

1. 用户首次打开应用，被要求设置一个 **主密码 (Master Password)**。  
2. 客户端在本地生成一个唯一的随机盐值 (Salt)。  
3. 使用 **PBKDF2** (至少 100,000 次迭代) 结合主密码和盐值，衍生出一个 256 位的 **会话密钥 (Session Key)**。  
4. 该密钥用于 **AES-GCM 字段级加密/解密**（如密码、私钥等敏感字段），并只驻留在内存中的 `SessionKeyHolder`。
5. 客户端本地保存 **salt** 与 **password_hash**（用于校验主密码正确性），不保存明文主密码。

**优点：** 只要用户记得主密码，未来升级到 v2.0 时，就可以在另一台设备上使用相同的主密码和从云端同步来的盐值，重新生成相同的密钥，从而解密数据。

### **2.2 生物识别解锁（便捷方案）**
- 生物识别仅作为 **UI 门禁**（BiometricPrompt 成功即放行）
- 开启时将派生密钥 **Base64 保存到本地偏好**
- 解锁时读取该密钥并写入 `SessionKeyHolder`
- **不绑定 AndroidKeyStore CryptoObject**，以稳定性优先

## **3\. 技术栈选型 (v1.0)**

* **开发语言：** Kotlin。  
* **UI 框架：** Jetpack Compose。  
* **架构模式：** MVVM。  
* **本地数据库：** Room（敏感字段使用 **AES-GCM** 手动加密）。  
* **SSH/SFTP 协议库：** JSch。  
* **Terminal View：** 基于 xterm.js 终端内核。

## **4\. 数据模型 (v1.0 Database Schema)**

数据库使用 Room 存储，**敏感字段采用 AES-GCM 手动加密**（如 password/key/passphrase），明文字段仅保留必要信息（alias/host/username/port）。

### **VpsEntity 表 (Room Entity)**

| 字段名 | 类型 | 描述 |
| :---- | :---- | :---- |
| id | Long | 主键，自增 |
| alias | String | 服务器别名 |
| host | String | IP 或域名 |
| port | Int | 端口，默认 22 |
| username | String | 登录用户名 |
| authType | String | "PASSWORD" 或 "KEY" |
| encryptedPassword | String? | SSH 密码（AES-GCM 密文） |
| encryptedKeyContent | String? | 私钥文件内容（AES-GCM 密文） |
| encryptedKeyPassphrase | String? | 私钥短语（AES-GCM 密文，可选） |
| createdAt | Long | 时间戳 |
| updatedAt | Long | 时间戳（为 v2.0 的冲突检测做准备） |

## **5\. 开发路线图 (v1.0)**

1. **阶段 1：项目初始化与安全基础**  
   * 设置主密码界面、盐值生成、PBKDF2 衍生密钥逻辑。  
   * 配置 Room + 字段级加密，确保敏感字段只能在正确主密码下解密。  
   * 实现生物识别应用锁。  
2. **阶段 2：VPS 列表管理 (CRUD)**  
   * 实现 VPS 信息的增删改查 UI。  
3. **阶段 3：SSH 终端集成**  
   * 集成 SSH 库和终端 View。  
   * 实现密码和密钥连接逻辑。  
4. **阶段 4：SFTP 功能**  
   * 实现文件浏览 UI。  
   * 实现文件操作逻辑（上传、下载、删除等）。  
5. **阶段 5：打磨与发布**  
   * 完善 UI/UX，处理报错。

## **6\. 当前进度与遗留问题（debug-10）**

**已完成模块：**
- 主密码 + PBKDF2 派生密钥
- AES-GCM 字段级加密
- 生物识别便捷解锁（UI 门禁方案）
- VPS CRUD
- SSH 连接（JSch）
- SFTP 基本功能
- 备份/恢复（Downloads + 加密包）
- 应用内 Debug Log

**遗留问题：**
- **终端输入体验不稳定**：
  - IME 弹出遮挡显示区域
  - Backspace 兼容性不足
  - 光标与输入位置无法准确同步

**推荐优化方向：**
- 使用成熟终端组件（xterm/VT100/terminal-emulator）代替文本拼接方案
- 让终端层接管光标、控制序列、输入事件和滚动

## **7\. 实践经验总结**

- **JSch 写入必须在 IO 线程**，否则容易出现无响应。
- **Compose + 隐藏输入框方案有上限**：IME 遮挡、光标定位、控制键处理复杂且不稳定。
- **快捷键方案需依赖终端层支持**：仅靠 raw 字符注入无法保证所有按键行为。

**第二阶段：VMan v2.0 (云同步扩展版)**

**目标：** 在 v1.0 的基础上，引入远程后端和端到端加密 (E2EE) 同步功能。

## **1\. 核心功能規格 (v2.0 新增)**

* **用户账户系统：** 在后端注册/登录。  
* **云同步：** 将本地加密后的服务器数据同步到远程服务器。  
* **端到端加密 (E2EE)：** **核心安全原则：主密码和解密后的明文数据绝对不能发送给服务器。** 服务器仅存储密文。

## **2\. 安全架构升级 (v1.0 \-\> v2.0 平滑升级)**

### **2.1 账户密码与主密码**

为了获得更好的体验，我们建议将登录后端的密码与加密数据的“主密码”分开。

* **账户密码 (Account Password)：** 仅用于访问 API（登录）。  
* **主密码 (Master Password)：** v1.0 中定义的那个，用于生成数据加密密钥。**它仍然只存在于客户端内存中。**

### **2.2 同步流程设计 (端到端加密)**

由于 v1.0 使用字段级加密，v2.0 可直接复用加密字段同步，但仍需完善同步与冲突处理策略：

* **本地存储调整：**  
  * 将 Room 模型调整为 v2.0 文档中的形式（增加 encryptedData 字段）。  
  * 数据离开本地 Room 数据库之前，使用 **主密码衍生的密钥 (DEK)** 对密码/密钥内容进行 **AES-GCM** 手动加密，然后存入 encryptedData 字段。  
  * 本地只需保留字段级加密即可（敏感字段不出明文），无需额外的全库加密。  
* **上传流程 (同步)：**  
  1. 客户端整理准备同步的条目。  
  2. 确认数据已使用 DEK 手动加密。  
  3. 通过 HTTPS 发送加密后的 JSON 对象给后端。  
* **下载流程 (同步)：**  
  1. 从后端 HTTPS 获取加密的数据包。  
  2. 提示用户输入主密码（如果不在内存中）衍生出 DEK。  
  3. 使用 DEK 解密数据包，加载到本地。

### **2.3 升级体验**

用户从 v1.0 升级到 v2.0 后：

1. **第一步：迁移数据。** 客户端提示用户输入主密码，衍生 DEK 解密本地字段后，再按 E2EE 逻辑重新加密，写入新的数据模型。  
2. **第二步：注册账户。** 用户注册后端账户，同时上传自己的**加密盐值**。  
3. **第三步：开始同步。**

## **3\. 技术栈选型 (v2.0 新增)**

* **后端服务：** Python (FastAPI), Go (Gin), 或 Node.js。  
* **后端数据库：** PostgreSQL。  
* **网络框架：** Retrofit2 (安卓客户端)。  
* **加密库：** KotlinX Crypto (安卓客户端手动加密)。

## **4\. 数据模型调整 (v2.0)**

### **客户端新的 Room Entity: VpsEntity\_v2**

| 字段名 | 类型 | 描述 |
| :---- | :---- | :---- |
| id | Long | 本地主键 |
| remoteId | Long? | 远程后端主键 |
| alias | String | 明文 |
| username | String | 明文 |
| authType | String | "PASSWORD" 或 "KEY" |
| encryptedData | String | **核心：包含加密后的密码或私钥内容的 JSON 密文 (AES-GCM)** |
| updatedAt | Long | 用于冲突检测 |

## **5\. 结论**

通过从 v1.0 就引入“主密码+PBKDF2”的安全模型，我们为未来的 v2.0 升级奠定了坚实的基础。V1.0 专注于做一个好用、安全的本地工具，而 v2.0 则专注于打破设备局限，实现安全的数据同步。这不仅节省了初期的开发成本，也降低了安全设计的复杂度。
